# Роль и контекст

**Проект:** React Native приложение (Expo SDK 54) с mock API  
**Стек:** TypeScript, React Navigation, Zustand, React Query  
**Платформы:** iOS, Android

# Структура тест-кейсов

Создай файл `test-cases.md` со следующей структурой:

## 1. Общая информация
- Название проекта
- Версия приложения
- Дата создания тест-кейсов
- Автор
- Окружение тестирования (устройства, ОС)

## 2. Матрица трассировки
Таблица соответствия: User Story → Тест-кейсы

## 3. Тест-кейсы по фичам

### Формат каждого тест-кейса:

```markdown
### TC-[ID]: [Название тест-кейса]

**Приоритет:** High / Medium / Low  
**Предусловия:** [Что должно быть настроено до теста]  
**Шаги:**
1. [Шаг 1]
2. [Шаг 2]
3. [Шаг 3]

**Ожидаемый результат:** [Что должно произойти]  
**Фактический результат:** [Заполняется при выполнении]  
**Статус:** Pass / Fail / Blocked  
**Связанные требования:** [FR-*, US-*, SCR-*]

## Инструкции по выполнению
1.	Изучи аналитику: Прочитай все указанные документы для понимания требований
2.	Создай файл: test-cases.md в корне проекта
3.	Заполни структуру: Общая информация, матрица трассировки, тест-кейсы
4.	Детализируй шаги: Каждый шаг должен быть конкретным и воспроизводимым
5.	Добавь приоритеты: High — критичный функционал, Medium — важный, Low — второстепенный
6.	Укажи связи: Каждый тест-кейс должен ссылаться на требования (FR, US, SCR)
Критерии качества тест-кейсов
•	Полнота: Все требования из аналитики покрыты тест-кейсами
•	Атомарность: Каждый тест-кейс проверяет одну конкретную функцию
•	Воспроизводимость: Шаги понятны и могут быть выполнены любым тестировщиком
•	Независимость: Тест-кейсы не зависят друг от друга (кроме явно указанных предусловий)
•	Трассировка: Каждый тест-кейс ссылается на требования
•	Приоритизация: Тест-кейсы имеют приоритеты для планирования тестирования
