# Варианты использования (Use Cases)

**Проект:** Мобильное приложение «Вертикаль» (Клиент)  
**Версия:** 1.0  

---

## UC-01: Запись на тренировку

**Актёры:** Клиент, Приложение, API  
**Предусловия:** Клиент авторизован в приложении.  
**Триггер:** Клиент нажал кнопку «Записаться» на экране деталей слота.

### Основной поток
1. Клиент открывает экран деталей слота.
2. Система отображает информацию о слоте, свободные места и доступность проката.
3. Клиент нажимает «Записаться».
4. Система отображает форму выбора снаряжения (своё / прокат).
5. Клиент выбирает опцию проката и размер скальников (если нужно).
6. Система рассчитывает итоговую стоимость и отображает экран подтверждения.
7. Клиент нажимает «Подтвердить бронь».
8. Приложение генерирует `Idempotency-Key` и отправляет POST-запрос на `/api/bookings`.
9. API проверяет доступность мест и прокатного фонда, создаёт бронь.
10. API возвращает статус 201 Created с данными брони.
11. Система показывает экран успеха «Вы записаны!» и сохраняет бронь в локальный кэш.
12. Система планирует локальное напоминание (push) за 2 часа до начала.

### Альтернативные потоки
- **A1. Клиент не авторизован.**
  - Шаг 3: Система перенаправляет на экран авторизации (UC-03). После успешной авторизации возвращает на шаг 4.
- **A2. Клиент заблокирован (много no-show/поздних отмен).**
  - Шаг 3: Система проверяет статус клиента. Если `blocked_until > now`, показывает экран блокировки с датой разблокировки. Флоу прерывается.

### Исключения и ошибки
- **E1. Слот уже занят (409 Conflict).**
  - Шаг 9: API возвращает 409. Приложение показывает тост: «Места закончились, пока мы оформляли бронь». Клиент возвращается к списку слотов.
- **E2. Недостаточно прокатного снаряжения (422 Unprocessable Entity).**
  - Шаг 9: API возвращает 422 с кодом `RENTAL_GEAR_UNAVAILABLE`. Приложение показывает ошибку: «Нужный размер скальников закончился. Попробуйте без проката или выберите другой слот».
- **E3. Сетевая ошибка / Таймаут (5xx / Timeout).**
  - Шаг 9: Запрос не прошёл. Приложение показывает ошибку: «Не удалось связаться с сервером. Проверьте интернет». Кнопка «Повторить» использует тот же `Idempotency-Key`.

### Матрица ошибок

| Код | HTTP | Обязательные details | UX-реакция | Retry |
|-----|------|----------------------|------------|-------|
| `SLOT_FULL` | 409 | `slot_id` | Тост: «Места закончились». Возврат к списку. | Нет |
| `RENTAL_GEAR_UNAVAILABLE` | 422 | `missing_sizes: [42, 43]` | Alert: «Нет размеров...». Предложить убрать прокат. | Нет |
| `BOOKING_DEADLINE_PASSED` | 400 | `deadline: "2026-07-04T17:00:00Z"` | Alert: «Запись закрыта за 30 мин до начала». | Нет |
| `CLIENT_BLOCKED` | 403 | `blocked_until: "2026-07-10"` | Экран блокировки с датой. | Нет |
| `NETWORK_ERROR` | - | - | Тост: «Нет соединения». Кнопка Retry. | Да (с тем же Idempotency-Key) |
| `SERVER_ERROR` | 500 | - | Тост: «Ошибка сервера. Попробуйте позже». | Да (с тем же Idempotency-Key) |

### Постусловия
- Бронь создана на сервере.
- Бронь отображается в разделе «Мои записи».
- Запланировано push-уведомление.

## UC-02: Отмена брони

**Актёры:** Клиент, Приложение, API  
**Предусловия:** У клиента есть активная бронь со статусом `Confirmed`.  
**Триггер:** Клиент нажал «Отменить» в деталях брони.

### Основной поток
1. Клиент открывает детали активной брони.
2. Система проверяет время до начала слота.
3. Если до начала > 2 часов: Система показывает диалог подтверждения «Отменить бронь?».
4. Клиент подтверждает отмену.
5. Приложение отправляет DELETE-запрос на `/api/bookings/{id}` с `Idempotency-Key`.
6. API удаляет бронь, освобождает место и прокатный фонд.
7. API возвращает 200 OK.
8. Система обновляет статус брони на `Cancelled` в UI и показывает тост «Бронь отменена».

### Альтернативные потоки
- **A1. Поздняя отмена (до начала < 2 часов).**
  - Шаг 2: Система определяет, что дедлайн пройден.
  - Шаг 3: Система показывает диалог с предупреждением: «Отмена менее чем за 2 часа. Это будет засчитано как поздняя отмена. 3 таких отмены приведут к блокировке на 7 дней».
  - Шаг 4: Клиент подтверждает. Флоу продолжается с шага 5.

### Исключения и ошибки
- **E1. Бронь уже отменена или завершена (410 Gone).**
  - Шаг 6: API возвращает 410. Приложение обновляет UI и показывает: «Эта бронь уже неактивна».
- **E2. Бронь отменена скалодромом.**
  - Шаг 2: Система видит статус `Cancelled_By_Organizer`. Кнопка «Отменить» скрыта. Показывается причина отмены.

### Матрица ошибок

| Код | HTTP | Обязательные details | UX-реакция | Retry |
|-----|------|----------------------|------------|-------|
| `BOOKING_NOT_FOUND` | 410 | `booking_id` | Тост: «Бронь не найдена или уже неактивна». | Нет |
| `BOOKING_ALREADY_CANCELLED` | 409 | `current_status` | Тост: «Бронь уже отменена». | Нет |
| `ORGANIZER_CANCELLED` | 403 | `reason` | UI: показ причины отмены организатором. Кнопка отмены скрыта. | Нет |

### Постусловия
- Бронь имеет статус `Cancelled` (или `Late_Cancelled` на сервере).
- Место и снаряжение освобождены.
- Счётчик поздних отмен клиента увеличен (если применимо).

## UC-03: Авторизация по OTP

**Актёры:** Клиент, Приложение, API, SMS-шлюз  
**Предусловия:** Клиент открыл приложение впервые или вышел из аккаунта.  
**Триггер:** Клиент ввёл номер телефона на экране входа.

### Основной поток
1. Клиент вводит номер телефона в формате +7 (XXX) XXX-XX-XX.
2. Приложение валидирует формат номера.
3. Приложение отправляет POST-запрос на `/api/auth/otp` с номером телефона.
4. API генерирует OTP, отправляет SMS и возвращает 200 OK с `otp_id` и TTL (5 мин).
5. Приложение переходит на экран ввода кода и запускает таймер обратного отсчёта (60 сек до повторной отправки).
6. Клиент вводит 6-значный код из SMS.
7. Приложение отправляет POST-запрос на `/api/auth/verify` с `otp_id` и кодом.
8. API проверяет код, создаёт сессию, возвращает 200 OK с JWT (access + refresh).
9. Приложение сохраняет токены в защищённое хранилище и переходит на главный экран (список слотов).

### Альтернативные потоки
- **A1. Неверный код.**
  - Шаг 7: API возвращает 401 Unauthorized с кодом `INVALID_OTP`. Приложение показывает ошибку под полем ввода и очищает поле.
- **A2. Истёк срок действия OTP.**
  - Шаг 7: API возвращает 410 Gone с кодом `OTP_EXPIRED`. Приложение показывает тост «Код устарел» и возвращает на ввод номера.
- **A3. Превышен лимит запросов (Rate Limit).**
  - Шаг 3: API возвращает 429 Too Many Requests. Приложение показывает таймер блокировки номера (например, 15 минут).

### Матрица ошибок

| Код | HTTP | Обязательные details | UX-реакция | Retry |
|-----|------|----------------------|------------|-------|
| `INVALID_PHONE` | 400 | `format` | Inline-ошибка: «Неверный формат номера». | Нет |
| `INVALID_OTP` | 401 | `attempts_left` | Inline-ошибка: «Неверный код. Осталось попыток: N». | Нет |
| `OTP_EXPIRED` | 410 | - | Тост: «Срок действия кода истёк». Возврат к вводу номера. | Нет |
| `RATE_LIMIT_EXCEEDED` | 429 | `retry_after: 900` | Alert: «Слишком много запросов. Попробуйте через 15 мин». | Нет |

### Постусловия
- Клиент авторизован.
- Токены сохранены локально.
- Загружен профиль клиента и расписание.
