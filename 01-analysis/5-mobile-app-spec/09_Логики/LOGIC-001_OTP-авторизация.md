## 2. Входные данные
- `phone` (String): Номер телефона в формате E.164 (+7...).
- `otp_code` (String): 6-значный код из SMS.
- `otp_id` (String): Идентификатор сессии OTP, полученный при запросе кода.

---

## 3. Обзор
**User Story:** US-001 — Быстрый вход без пароля.  
**Бизнес-ценность:** Снижение порога входа, отсутствие нагрузки на поддержку по восстановлению паролей.

## 5. Флоу
flowchart TD
    A[Ввод номера] --> B{Валиден?}
    B -->|Нет| C[Inline ошибка]
    B -->|Да| D[POST /auth/otp]
    D --> E{Успех?}
    E -->|Да| F[Запуск таймера 60с, экран ввода кода]
    E -->|429| G[Блокировка на 15 мин]
    F --> H[Ввод кода]
    H --> I[POST /auth/verify]
    I --> J{Успех?}
    J -->|200| K[Сохранение токенов в Keychain/Keystore]
    J -->|401| L[Inline ошибка, очистка поля]
    J -->|410| M[Тост 'Код устарел', возврат к номеру]

## 6. API-запросы
## 6.1. Запрос OTP
Метод: POST
Путь: /api/v1/auth/otp
Body: { "phone": "+79001234567" }
Обработка: 200 (успех), 400 (неверный формат), 429 (rate limit).
6.2. Проверка OTP
Метод: POST
Путь: /api/v1/auth/verify
Body: { "otp_id": "...", "code": "123456" }
Обработка: 200 (токены), 401 (неверный код), 410 (истек TTL).

## 7. Локальное хранение
Хранилище: Keychain (iOS) / EncryptedSharedPreferences (Android).
Ключи: access_token, refresh_token, token_expires_at.
Тип: Secure Storage.

## 8. Связанные требования
FR-001, FR-002, FR-003, FR-004
NFR-007 (JWT), NFR-008 (Rate-limit)

## 9. Критерии приёмки
Given: Пользователь ввел номер +79001234567. When: Нажал «Получить код». Then: Отправлен POST /auth/otp, запущен таймер.
Given: Таймер истек. When: Нажал «Отправить снова». Then: Повторный запрос, таймер сброшен.
Given: Введен неверный код. When: Нажал «Подтвердить». Then: Получен 401, показана ошибка, поле очищено.

## 10. Обработка ошибок
Сетевые ошибки: Показ тоста, возможность повторить запрос.
429 Too Many Requests: Полная блокировка UI ввода на время, указанное в retry_after.
