## 1. Трассировка требований

Проверка полноты связей между артефактами:

- **FR → Источник:** Все функциональные требования (FR-001 — FR-029) имеют ссылку на источник (бриф, R-NNN или аналитическое решение). Пробелов нет.
- **UC → FR:** Все Use Cases (UC-01, UC-02, UC-03) полностью покрываются функциональными требованиями. Матрицы ошибок в UC согласуются с кодами ошибок в API-спецификации.
- **US → FR:** Все 15 User Stories (US-001 — US-015) имеют прямую привязку к FR. Критерии приёмки (Given/When/Then) проверяемы.
- **SCR/BS → FR/NFR:** Каждый экран и Bottom Sheet в папке `5-mobile-app-spec` содержит раздел «Связанные требования», где указаны FR и NFR.
- **LOGIC → FR:** Все 8 логик (`LOGIC-001` — `LOGIC-008`) ссылаются на соответствующие FR.
- **API → SCR:** `operationId` в ТЗ экранов (например, `requestOtp`, `createBooking`) полностью совпадают с `operationId` в файлах `api/*/api.yaml`.

**Вывод по трассировке:** Трассировка выстроена корректно, разрывов связей не обнаружено.

## 2. Полнота

- **Покрытие UC экранами:** Все сценарии из Use Cases (успешный флоу, альтернативы, ошибки) отражены в ТЗ экранов (SCR-001, SCR-003, SCR-004, SCR-006) и логиках.
- **Покрытие FR:** Каждое FR имеет хотя бы одну точку реализации (экран, логику или API-эндпоинт).
- **Покрытие NFR:** Нефункциональные требования имеют проверяемые критерии (например, NFR-003 — время загрузки < 2с, NFR-008 — rate-limit 5 запросов/час).
- **Потерянные требования:** Не обнаружено. Все идеи из брифа Оли (оценки, прокат, санкции, уведомления) нашли отражение в FR или были явно вынесены в Phase 2 (онлайн-оплата).

**Вывод по полноте:** Документация покрывает весь заявленный скоуп MVP.

## 3. Непротиворечивость

- **Числа и лимиты:** 
  - Дедлайн отмены (2 часа) везде одинаков: FR-020, LOGIC-004, UC-02, BS-003.
  - Дедлайн записи (30 минут) везде одинаков: FR-017, SCR-003, LOGIC-002.
  - Размеры скальников (36-45) согласованы в Data Model, SCR-004 и API Spec.
- **Статусы сущностей:** 
  - Статусы `Booking` (`CONFIRMED`, `CANCELLED`, `LATE_CANCELLED`, `CANCELLED_BY_ORGANIZER`, `COMPLETED`) идентичны в `4-design/data-model.md` и `api/bookings/models.yaml`.
  - Статусы `Slot` (`SCHEDULED`, `CANCELLED_BY_ORGANIZER`) идентичны.
- **Противоречия между Дизайн-брифом и ТЗ:** Расхождений не выявлено. Паттерн состояний (Loading/Content/Empty/Error) из `00-foundations.md` единообразно применён во всех SCR.

**Вывод по непротиворечивости:** Данные и бизнес-правила консистентны across all documents.

## 4. Цифровая гигиена (Checklist)

- [x] **Нет выдуманных интеграций:** Используются только существующий бэкенд, SMS-шлюз (абстрактно) и Firebase/APNs.
- [x] **Глоссарий:** Все ключевые сущности (Слот, Бронь, No-show, Дедлайн) описаны в `domain-description.md` и `feature-list.md`.
- [x] **ФТ и НФТ не перепутаны:** Требования к UI/бизнес-правилам во FR, требования к производительности/безопасности в NFR.
- [x] **Уникальность ID:** Все ID (BR, FR, NFR, UC, US, SCR, BS, LOGIC, R) уникальны в рамках своих префиксов.
